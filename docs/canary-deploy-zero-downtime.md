# 카나리 무중단 배포 — 다운타임 5~10분→무중단, 자동 롤백

## 문제

DDD 기반 마이크로서비스 아키텍처로 9개 이상의 서비스를 운영하는데, 모든 서비스를 일괄 빌드/배포하고 있었다. 코드 한 줄 바꿔도 9개 서비스를 전부 빌드하니 배포에 13분 31초가 걸렸고, 약 20분간 개발이 중단되었다. 배포 시간을 줄인 뒤에도 docker-compose down → up 과정에서 5~10분의 다운타임이 발생했고, 실제 고객으로부터 서비스 장애 문의가 들어왔다.

## 분석

| 항목 | 내용 |
| --- | --- |
| 아키텍처 | Spring Boot 기반 마이크로서비스 (9개 서비스) + React 프론트엔드 |
| 배포 전략 | 카나리(Canary) 배포 - 7단계 파이프라인 |
| CI/CD | GitHub Actions → Docker Build → SCP → EC2 카나리 배포 |
| 서비스 디스커버리 | Netflix Eureka |
| 모니터링 | Prometheus + Loki + Grafana |

배포 전략 선정 비교.

| 전략 | 장점 | 단점 | 미선택 이유 |
| --- | --- | --- | --- |
| 블루/그린 | 즉시 전환, 완벽한 롤백 | 2배의 인프라 필요 | EC2 1대 환경에서 메모리 2배 감당 불가 |
| 롤링 | 점진적 배포, 리소스 효율적 | 롤백 복잡, 점진적 테스트 어려움 | 문제 시 어디까지 롤백할지 판단이 어려움 |
| 카나리 | 검증 후 전환, 빠른 롤백, 리스크 최소화 | 구현 복잡도 | 채택 |

카나리를 선택한 핵심 이유 3가지.

1. 기존 인프라 활용. Eureka + API Gateway 구조가 이미 있어서, 같은 서비스명으로 카나리 인스턴스를 등록하면 Eureka가 자동으로 라운드로빈 로드밸런싱을 수행한다. Kubernetes나 별도 로드밸런서 없이 Spring Cloud만으로 구현 가능했다.
2. 리소스 효율성. 구버전(512MB) + 카나리(512MB) = 약 1GB로, 배포 중 일시적으로만 리소스를 더 사용하고 전환 후 원래대로 돌아온다.
3. 즉시 롤백. 문제 발생 시 카나리 컨테이너만 제거하면 3초 내 롤백이 완료되고, 구버전이 그대로 서비스를 계속한다.

## 해결

### 1차 개선 — 선별 빌드로 배포 시간 단축

GitHub Actions에서 `git diff --name-only origin/main...HEAD`로 변경된 파일 목록을 추출한다. `fetch-depth: 0`으로 전체 커밋 히스토리를 확보한 뒤, 각 서비스 디렉토리에 변경이 있는지 체크한다. 변경된 서비스만 Gradle/Maven 빌드 → Docker 이미지 생성 → `.tar` 파일 저장을 수행하고, `compgen -G "*.tar"`로 실제 생성된 이미지만 EC2로 전송한다.

### 2차 개선 — 카나리 배포 파이프라인

7단계 파이프라인으로 구성된다.

| Phase | 이름 | 설명 | 실패 시 |
| --- | --- | --- | --- |
| 1 | Docker 이미지 확인 | docker load로 로드된 이미지에 canary 태그 부여 | 배포 중단 |
| 2 | 카나리 인스턴스 시작 | 기존 포트 + 1000번으로 신버전 컨테이너 실행 (같은 Docker 네트워크) | 배포 중단 |
| 3 | 헬스체크 | /actuator/health를 10초 간격, 최대 12회(2분) 확인 | 자동 롤백 |
| 4 | Eureka 등록 확인 | 컨테이너 로그에서 등록 메시지 확인 (discovery-service는 스킵) | 자동 롤백 |
| 5 | 트래픽 모니터링 | 1분간 10초 간격 6회 헬스 재검증 (Eureka 라운드로빈 50:50 트래픽 분배) | 자동 롤백 |
| 6 | 구버전 제거 | docker-compose stop/rm으로 기존 컨테이너 Graceful Shutdown | - |
| 7 | 정식 전환 | 카나리 이미지를 latest로 태그 변경, docker-compose up -d 재시작 | - |

핵심 원리. Eureka에 같은 서비스명으로 구버전과 카나리가 동시 등록되면, 자동으로 라운드로빈 50:50 트래픽 분배가 이루어진다. 검증이 끝나면 구버전을 Graceful Shutdown으로 안전하게 내리고 카나리를 정식으로 전환한다.

### CI/CD 파이프라인 전체 구조

```
[GitHub PR → main] → [변경 감지] → [선별 빌드] → [EC2 전송] → [배포 판단] → [카나리 or Fresh]
```

1. 변경 감지. `git diff --name-only origin/main...HEAD`로 변경 파일 목록 추출. `fetch-depth: 0`으로 전체 커밋 히스토리 확보. 9개 서비스 각각 변경 여부를 개별 체크.
2. 선별 빌드. 변경된 서비스만 Gradle/Maven 빌드 → Docker 이미지 → `.tar` 저장. GitHub Secrets로 `.env` 파일 동적 생성.
3. EC2 전송. 기존 `.tar` 파일 삭제 후 새 이미지만 SCP 전송. docker-compose.yml, deploy-canary.sh, 모니터링 설정도 함께 전송.
4. 배포 판단. EC2에서 `docker-compose ps`로 실행 중인 서비스 확인. 이미 실행 중이면 카나리 배포(deploy-canary.sh), 신규 서비스면 Fresh 배포(docker-compose up -d).
5. 모니터링 인프라. 배포 완료 후 Loki, Grafana 항상 실행 상태 유지.

### 자동 롤백

| 시점 | 트리거 조건 | 롤백 동작 |
| --- | --- | --- |
| Phase 3 (헬스체크) | 2분 내 /actuator/health가 UP이 안 됨 | 카나리 stop/rm, canary 이미지 삭제, 로그 80줄 출력 |
| Phase 4 (Eureka 등록) | 1분 내 Eureka 등록 메시지 없음 | 카나리 제거, discovery-service 로그/상태도 출력 |
| Phase 5 (트래픽 모니터링) | 모니터링 중 헬스 상태 DOWN 감지 | 카나리 즉시 제거 |

모든 경우에 구버전 컨테이너는 건드리지 않는다. 롤백 = 카나리만 제거하면 끝이므로 약 3초 내 완료된다. 롤백 시 디버깅을 위해 카나리 컨테이너의 최근 로그 80줄을 자동 출력하고, Eureka 등록 실패 시에는 discovery-service 로그와 health 상태까지 함께 출력한다.

### 프론트엔드와 백엔드 배포 차이

| 구분 | 백엔드 (Spring Boot) | 프론트엔드 (React + Nginx) |
| --- | --- | --- |
| 헬스체크 엔드포인트 | /actuator/health | /health |
| 헬스체크 대기 시간 | 최대 2분 (12회 x 10초) | 최대 1분 (6회 x 10초) |
| Eureka 등록 확인 | Phase 4에서 로그 파싱으로 확인 | Phase 4 스킵 (프론트엔드이므로 불필요) |
| 빌드 방식 | Gradle/Maven → JAR → Docker | npm run build → Multi-stage Docker (Node → Nginx) |
| CI 트리거 | PR to main | Push to main |

프론트엔드 배포 경량화. 백엔드 카나리 배포 스크립트를 프론트엔드에 맞게 경량화했다. Nginx 정적 파일 서빙이라 Eureka 서비스 등록과 1분간 트래픽 모니터링 단계가 불필요하여 제거하고, 배포 Phase를 7단계에서 5단계로 줄였다. 기존 배포 스크립트에서 구버전과 카나리 컨테이너가 동시에 중단되는 구간이 있어 다운타임이 발생했는데, 이미지 태그를 먼저 변경한 뒤 docker-compose up --force-recreate로 즉시 교체하는 방식으로 변경하여 다운타임 제로를 달성했다. CI/CD 파이프라인에서는 Docker 내부에서 npm install과 빌드를 수행하던 방식을 GitHub Actions의 Node.js 캐싱으로 전환하여, 의존성 변경이 없는 일반 코드 배포 시 npm install 단계를 캐시로 스킵하도록 했다. 배포 시간을 기존 약 4분 20초에서 1분 30초 내외로 단축했다.

### 트러블슈팅

Eureka 등록 타이밍 문제. Spring Boot의 `/actuator/health`가 UP을 반환하는 시점과 Eureka에 실제 등록이 완료되는 시점 사이에 시간차가 있었다. 헬스체크만 통과하면 바로 전환했더니, API Gateway가 카나리 인스턴스를 아직 모르는 상태에서 구버전을 내려버려서 요청이 실패했다. Phase 4를 추가해 컨테이너 로그에서 `registered with eureka`, `registration status: 204` 패턴을 5초 간격으로 확인하도록 했다. Eureka 서버 자체가 안 떠있을 수도 있으므로, discovery-service 헬스를 먼저 확인(최대 2분)한 후 등록 체크를 진행하도록 2단계로 분리했다.

CGroup v2 호환성 버그. JDK 17.0.12 이상에서 CGroup v2 관련 버그로 컨테이너가 기동되지 않는 현상이 발생했다. `JAVA_TOOL_OPTIONS=-XX:-UseContainerSupport` 옵션을 추가해 컨테이너 리소스 자동 감지를 비활성화하여 해결했다.

컨테이너 네트워크 감지 실패. 카나리 컨테이너를 실행할 때 기존 서비스의 네트워크를 정확히 파악하지 못해 서비스 간 통신이 안 되는 문제가 있었다. `docker inspect --format` 명령으로 기존 컨테이너의 네트워크를 동적으로 감지하도록 수정했다.
