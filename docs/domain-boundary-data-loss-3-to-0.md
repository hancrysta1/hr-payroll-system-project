# 도메인 경계 재설정으로 데이터 누락율 3.44% → 0%, MAU 71.5% 증가

## 문제

B2B로 사업장의 근무 기록과 그에 따른 급여를 대신 관리하는 서비스에서, 금전 정산과 직결되기 때문에 오차와 누락이 허용되지 않는 도메인이었다. 초기 설계에서는 확장성과 독립 배포를 고려해 스케줄 서비스와 급여 서비스를 분리하고, Outbox/Kafka 이벤트 기반으로 연결했다. 스케줄 생성 시 Outbox로 이벤트를 남기고, Kafka로 발행하면 급여 서비스가 이를 소비해 급여 계산 최소 단위를 생성/갱신하는 구조였다.

운영에서 급여 서비스가 비즈니스 규칙 위반, 특정 케이스 예외, 배포 순간의 네트워크 불안정 등으로 계산에 실패하면 재시도 횟수를 넘어 DLQ로 상태가 분리되었다. 스케줄은 생성되었는데 급여 최소 단위 데이터가 없거나 미완성인 상태가 남고, 이벤트는 DLQ에 격리되어 운영자가 수동 개입해야 했다. 정합성 실패(누락) 비율 3.44%가 발생했다.

## 분석

스케줄-급여의 도메인 특성상 "결과적 일관성"으로는 부족하고, 강한 일관성이 필요했다.

- 스케줄 수정됨 → 급여가 즉시 바뀌어야 함
- 스케줄 삭제됨 → 급여도 즉시 없어져야 함
- 스케줄만 있고 급여 최소 단위가 없는 상태는 업무적으로 허용되지 않음

급여 계산을 위한 최소 단위(하루)의 데이터가 스케줄과 1:1로 반드시 쌓여야 하는 불변조건이 있었다. "이벤트로 나중에 맞추자"는 설계가 근본적으로 위험했다.

Saga 트랜잭션(오케스트레이션/보상)이나 폴링 기반 재처리도 검토했지만, 배포 순간/네트워크 흔들림/타임아웃 같은 사각지대에서 한쪽만 성공한 고아 데이터(Orphan)가 발생할 수 있었다. 보상 로직이 항상 완벽히 동작해야 하는데, 급여 도메인은 잠깐의 불일치도 사용자 신뢰에 치명적이었다.

금융권/은행 시스템도 모든 기능을 한 덩어리로 만들지는 않지만, 잔액/원장처럼 금액 정합성이 핵심인 영역은 단일 원장과 강한 ACID 트랜잭션으로 불일치가 구조적으로 나지 않게 설계한다. 반면 알림/리포트/검색 같은 부가 기능은 비동기로 분리해도 된다. 이 맥락에서 스케줄-급여 최소 단위는 서비스의 "원장급 핵심"에 가까웠다.

기술적으로는 "서비스를 분리"했지만, 도메인 관점에서는 "하나의 거래(근무 → 급여)가 원자적으로 묶여야 하는 컨텍스트"였다. DDD에서 말하는 컨텍스트 경계가 잘못 설정되어 있었다.

## 해결

스케줄 서비스와 급여 서비스를 통합해 Workpay 서비스로 재설계했다.

- 동일 DB/동일 트랜잭션으로 schedule 생성과 payroll_day(급여 최소 단위) 생성을 1:1로 수행
- 하나라도 실패하면 전체 롤백
- 정합성은 이벤트/재시도에 의존하지 않고 DB 트랜잭션이 보장

Kafka는 정합성 핵심이 아니라, 알림/통계/부가 보고서 같은 비핵심 기능을 분리하는 용도로만 두었다.

## 결과

| 지표 | Before | After |
| --- | --- | --- |
| 정합성 실패(누락) 비율 | 3.44% | 0% |
| MAU | 기준값 | 71.5% 증가 |

안정성 확보 이후 실제 사용성이 개선되며 MAU 71.5% 증가로 이어졌다. 이 변화는 기능 추가 때문이 아니라, 고객이 "믿고 쓸 수 있는 상태"가 되었기 때문이었다.

> 금전 정산이 핵심인 B2B 도메인에서 이벤트 기반 분리 구조는 DLQ로 인해 정합성 공백을 구조적으로 남길 수 있음을 운영에서 체감했고, 도메인 불변조건(스케줄-급여 최소 단위 1:1)을 보장하기 위해 컨텍스트 경계를 재정의하여 Workpay로 통합, 단일 트랜잭션으로 정합성을 강제함으로써 누락율 3.44% → 0%, MAU 71.5% 증가를 달성했다.

### 추가 완성도 방향

- 정합성 실패율 3.44%를 어떻게 측정했는지(정의/지표)
- "누락/불일치"를 탐지하는 쿼리/모니터링 방식
- 전환 후 DB 제약조건(UNIQUE/FK)으로 1:1 강제한 설계
