# 모듈 분리를 통한 테스트 실행 최적화

> 원본 서비스(`Server`)의 `payroll-compute`(급여 계산 엔진) / `workpay-service`(근태·급여 서비스) **2-모듈 구조**에서 측정한 회고 문서.
> 본 포트폴리오 레포는 요약용 단일 모듈(`money-vo`)로 통합되어 있으며, 이 문서는 원본 서비스의 설계 결정 근거를 기록한 것.

## 측정 환경

- Gradle `--no-daemon --rerun-tasks` 옵션 (캐시 무효화 풀런 기준)
- 동일 머신, 동일 시점 측정

## 모듈별 테스트 실행 시간

| 모듈 | 실행된 태스크 | 실제 소요 시간(wall) | User CPU | System CPU | CPU 활용률 |
|---|---|---|---|---|---|
| payroll-compute (급여 계산 엔진) | 4 | **6.21초** | 13.72초 | 0.84초 | 234% |
| workpay-service (근태·급여 서비스) | 9 | **31.85초** | 68.28초 | 4.39초 | 228% |
| 합계(순차 실행 시) | 13 | 38.06초 | 82.00초 | 5.23초 | - |

두 모듈 모두 CPU 활용률 약 2.3코어 수준으로 병렬화 효율은 비슷하나, 태스크당 평균 실행 시간은 payroll-compute가 1.55초, workpay-service가 3.54초로 개별 테스트 자체의 무게가 다르다 (서비스 계층이 DB/통합 테스트 비중이 높기 때문).

## 분리를 통해 얻은 정량적 효과

### 1. 변경 영향 범위 축소 — 최대 약 84% 단축

급여 계산 엔진(payroll-compute)만 수정한 경우, 해당 모듈 테스트(6.21초)만 실행해도 검증이 완료된다. 분리 이전 단일 모듈 구조였다면 근태·급여 서비스까지 포함한 전체 38초 분량을 매번 돌려야 했다.

> 급여 계산 로직 변경 시: **38.06초 → 6.21초 (약 84% 단축)**

로컬 개발 중 "저장 → 테스트 → 확인" 루프가 6배 이상 빨라졌고, CI에서도 변경 모듈 한정 실행이 가능해졌다.

### 2. 모듈 병렬 실행 — 약 16% 단축

두 모듈에 걸친 변경이더라도 Gradle의 모듈 간 병렬 빌드로 동시 실행 가능하다.

- 순차 실행: 6.21 + 31.85 = **38.06초**
- 병렬 실행: max(6.21, 31.85) = **31.85초**

단일 모듈 구조였다면 내부적으로 병렬 실행할 수 있는 단위가 없어 해당 이점을 얻을 수 없다. 모듈 수가 늘어날수록 이 이점은 더 커진다.

### 3. 불필요한 재실행 제거 (증분 빌드)

변경되지 않은 모듈은 Gradle이 `UP-TO-DATE`로 판정해 테스트 태스크 자체를 건너뛴다. CI에서도 전체 재실행이 아닌 **변경 영향 모듈 한정 실행**이 자연스럽게 이뤄진다.

## 정성적 이점 (수치 외)

- **빌드 실패 격리**: 근태·급여 서비스의 컴파일 에러가 급여 계산 엔진 테스트 실행을 막지 않는다. 한 모듈의 장애가 전체 파이프라인을 중단시키지 않음.
- **의존성 경계 강제**: Gradle 모듈 간 의존 방향이 명시되어, 급여 계산 도메인과 근태 서비스 계층 사이의 순환 참조가 구조적으로 차단된다. 새로 합류하는 팀원도 의존 방향을 코드가 아닌 `build.gradle`의 `implementation project(...)` 한 줄로 파악할 수 있다.
- **책임 분리의 가시화**: 급여 계산이라는 순수 계산 로직과 근태·급여 서비스라는 상태 변경 로직이 패키지 수준이 아닌 **배포 가능 단위** 수준으로 분리되어, 계산 엔진을 다른 서비스에서 재사용하거나 독립 서비스로 분리할 여지가 생겼다.
- **테스트 특성에 맞는 전략 적용 가능**: 계산 엔진 모듈은 단위 테스트 중심으로, 서비스 모듈은 통합 테스트 중심으로 각각 다른 실행 전략(병렬도, 격리 수준)을 적용할 수 있다.

---

## 요약

모듈 분리는 단순히 "코드 정리"가 아니라 **실제 개발·CI 사이클의 반복 비용을 줄이는 구조적 장치**였다. 변경 영향 범위가 작을수록 효과는 더 크며 (급여 계산 로직만 수정 시 약 84% 단축), 구조적으로 의존 경계가 강제되고 장애 격리가 자연스럽게 이뤄진다는 정성적 이점까지 포함하면 유지보수 비용 측면에서도 명확한 개선이 있다.
