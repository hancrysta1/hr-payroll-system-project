# 급여 정산 시스템 설계로 금전 분쟁 문의 100% 제거

## 문제

운영 중인 알바 관리 서비스에서 급여 관련 문의가 반복 발생했음.

문의가 발생하는 근본 원인은 정산이 복잡하다는 것.
- 직원마다 급여 타입이 다름 (A직원은 시급제, B직원은 일급제, C직원은 월급제)
- 직원마다 수당 ON/OFF 설정이 다름 (야간/연장/휴일/주휴 각각)
- 설정이 ON이어도 조건이 충족되어야 실제 적용됨 (예: 주휴수당은 주 15시간 이상 근무해야 적용)
- 실제 문의 예시. "왜 주휴수당이 안 붙었냐" → 주 15시간 미만이라 조건 미충족인데, 사장님은 설정을 켰으니 당연히 적용될 거라 생각

기존 구조에는 여러 한계가 있었음.
- 최종 금액(`totalSalary`)만 반환 — 왜 이 금액인지, 왜 특정 수당이 미적용됐는지 알 수 없음
- 문의 발생 시 개발자가 직접 6~8개 DB 테이블을 추적 (건당 30분 이상)
- 설정은 현재값만 존재 — 확정 당시 어떤 기준이 적용됐는지 소급 불가

기존 서비스 레이어 코드에도 구조적 문제가 있었음.
- BigDecimal 연산이 서비스 곳곳에 산재 → scale/RoundingMode 불일치 시 1원 단위 계산 오차 발생 가능 (금전 시스템에서 치명적)
- 매직넘버(코드에 의미 없이 박혀있는 숫자) 하드코딩 (`480`, `12540`, `0.033`) → 의미를 모르면 수정 불가, 다른 개발자가 실수로 다른 값 사용 가능
- 계산 + I/O + 저장이 하나의 메서드에 혼합 → 계산 로직만 단위 테스트 불가능, Spring 컨텍스트 기동 필수 (수십 초)
- 정책이 코드에 박혀있음 → 수당 조건 변경 시 코드 수정 + 배포 필요, 런타임 변경 불가

## 해결

### 1. 정책의 DB 테이블화

코드에 하드코딩되어 있던 시급 결정 로직, 수당 조건, 공제 정책을 7개 DB 테이블로 분리했음.
서버 재배포 없이 직원별 설정을 런타임에서 변경 가능하게 됨.

| 기존 (코드 하드코딩)     | 변경 (DB 테이블)                                    |
|--------------------------|-----------------------------------------------------|
| 최저시급 상수값          | `minimum_wage` — 연도별 저장                        |
| 시급 고정                | `user_salary_wage` — 직원별 시급/일급/월급           |
| 수당 전체 일괄 적용      | `user_salary_allowances` — 직원별 ON/OFF            |
| 공제 전체 일괄 적용      | `user_salary_deductions` — 직원별 ON/OFF            |
| 보험 금액 고정           | `user_insurance_amount` — 직원별 커스텀 금액         |
| 특정일 시급 변경 불가    | `user_daily_wage` — 날짜별 시급 override             |
| 가산율 상수 50%          | `legal_allowance_policy` — 지점별 요율 설정          |

### 2. DDD(도메인 주도 설계) 패턴 적용 — 계산 로직 캡슐화

서비스 레이어에 산재된 BigDecimal 연산을 Value Object(값 객체)와 Domain Model(도메인 모델)로 캡슐화했음.

- Value Object 4개. `Money`(금액), `WorkMinutes`(근무시간), `Percentage`(비율), `HourlyWage`(통상시급)
- Domain Model. `DailySalaryCalculation` — 기본급 → 가산수당 → 세금 파이프라인을 하나의 도메인 모델에 응집
- 서비스 레이어에 산재된 BigDecimal 연산을 도메인 객체가 자기 규칙을 아는 구조로 전환함.
  - `Money.multiplyByMinutes(WorkMinutes)` — "시급 x 근무시간 = 급여"가 코드로 표현됨
  - `WorkMinutes.exceedsDailyLimit()` — 연장수당 조건 판정이 VO에 응집
  - `Money.applyTax(Percentage)` — 세금 계산 규칙이 금액 객체에 내장
  - Money끼리만 연산 가능, WorkMinutes 음수 불가 — 잘못된 사용이 컴파일 타임에 차단됨
- JPA/Spring 의존 없는 POJO — 도메인 규칙이 인프라에 묶이지 않고, `new`로 바로 검증 가능 (단위 테스트 20케이스, 3초)

DDD가 연산 오류를 원천 차단하는 구조는 다음과 같음.

기존 서비스 코드에서는 BigDecimal의 scale(소수점 자릿수)을 개발자가 매번 직접 지정해야 했음. 성능 문제가 아니라 정합성 문제 — 같은 원 단위 금액인데 개발자마다 scale을 다르게 넣으면 합산 시 소수점이 섞임.

```
// 기존 — 같은 원 단위 금액인데 scale이 다름
baseSalary = hourlyWage.multiply(minutes).divide(60, 0, HALF_UP);     // scale=0
nightAllowance = hourlyWage.multiply(rate).divide(60, 2, HALF_UP);    // scale=2
// 합산하면 10,030원 + 5,015.00원 → 소수점이 섞임
```

Money VO는 `final class`로 구현했음. final이므로 상속 불가 — 하위 클래스에서 연산 규칙을 오버라이드해서 깨뜨릴 수 없음. 생성자에서 `setScale(0, HALF_UP)`을 강제하므로 어디서 Money를 만들든 항상 원 단위 반올림됨. 개발자가 scale을 다르게 넣는 것 자체가 불가능.

테스트에 미치는 영향도 컸음. 기존에는 계산 로직이 서비스 레이어에 있어서 DB, Feign Client(서비스 간 HTTP 통신 라이브러리), Spring Security 등을 전부 띄워야 테스트 가능했음 (통합 테스트, 수십 초). Money/WorkMinutes 같은 VO와 DailySalaryCalculation 도메인 모델은 JPA/Spring 의존성이 없는 순수 Java이므로, new로 만들어서 바로 검증 가능. 20개 계산 케이스를 3초 내에 실행함.

매직넘버도 VO 상수로 전환했음.
- `480` → `WorkMinutes.DAILY_LIMIT` (일 8시간 = 480분)
- `12540` → `WorkMinutes.MONTHLY_DEFAULT` (월 209시간 = 12540분)
- `0.033` → `Percentage.TAX_RATE_3_3` (3.3% 원천징수)

코드만 봐서 480이 뭔지 몰랐던 것이, `DAILY_LIMIT`이라는 이름으로 의미가 명확해지고 다른 개발자가 실수로 다른 값을 사용할 수 없게 됨.

### 3. 계산 근거 구조화 + 스냅샷 보존

API 응답에 "왜 이 금액이 나왔는지" 계산 근거를 포함하고, 급여 확정 시 스냅샷으로 보존함.

시급 결정 근거 — 5단계 우선순위 체인 중 어디서 시급을 가져왔는지 기록.
```
① 특정일 시급 → ② 시급 변경 이력 → ③ 직원별 시급 → ④ 지점 기본 시급 → ⑤ 법정 최저시급
```

수당 적용/미적용 근거 — 설정 ON인데 미적용된 이유까지 기록.
```json
{
  "overtimeReason": {
    "settingEnabled": true,
    "applied": false,
    "notAppliedReason": "DAILY_LIMIT_NOT_EXCEEDED",
    "dailyThresholdMinutes": 480,
    "actualDailyMinutes": 420
  }
}
```

확정 시점 스냅샷 — 급여 확정 시 `salary_confirmation.reason_snapshot`에 JSON으로 저장. 적용된 설정값뿐 아니라 적용/미적용 사유까지 포함됨. 이후 설정이 변경되어도 확정 당시 "어떤 기준으로, 왜 이렇게 계산됐는지"를 누구나 확인 가능.

### 4. 캐싱 전략 — 변경 빈도별 TTL 분리

급여 계산 파이프라인에서 설정 조회 단계(시급 결정, 수당 확인, 공제 확인)에 Caffeine(JVM 로컬 캐시 라이브러리) 적용.

캐시 히트 조건. `cacheManager.getCache(캐시명).get(userId:branchId)`로 수동 조회함. 히트 시 DB 쿼리 스킵, 미스 시 DB 조회 후 캐시에 put. `@Cacheable` 대신 수동 조회를 사용한 이유는 캐시 미스 시 DB 조회 결과가 null이어도 캐시에 저장해야 하는 경우(설정이 없는 직원도 매번 DB를 치지 않도록)가 있고, 조회 로직 중간에 세밀한 제어가 필요했기 때문.

캐시 무효화. 사장님이 앱에서 직원 설정을 변경하는 시점에 `@CacheEvict(key = "userId:branchId")`로 해당 직원의 캐시만 즉시 삭제함. 예를 들어 "A직원 야간수당 OFF→ON 변경" → A직원 캐시만 무효화 → 다음 급여 계산 시 DB에서 새 설정 읽어서 캐시 갱신. 설정 변경이 아닌 경우 TTL(캐시 만료 시간) 만료까지 캐시 유지.

| 변경 빈도   | 캐시 수 | TTL    | 적용 단계              | 무효화 조건                  |
|-------------|---------|--------|------------------------|------------------------------|
| 거의 안 변함 | 4개     | 24시간 | 수당/공제 설정 조회     | 사장님이 설정 변경 시 evict  |
| 가끔 변함   | 3개     | 1시간  | 시급 결정, 가산수당 요율 | 시급 변경/정책 수정 시 evict |
| 집계 결과   | 1개     | 30분   | 급여 요약 조회          | TTL 만료                     |
| 매번 계산   | -       | -      | 근무시간/기본급/수당 계산 | 캐싱 안 함                  |

## 결과

왜 문의가 0건이 됐는가.
- "왜 주휴수당이 안 붙었냐" → 명세에 `notAppliedReason: "WEEKLY_HOURS_NOT_MET"` (주 15시간 미달) 표시 → 설정은 ON이지만 조건 미충족이라는 사유가 보이니 사장님이 납득
- "왜 이 금액이냐" → 명세에 시급 출처, 근무시간, 수당 항목별 금액, 공제 내역이 전부 포함 → 개발자 추적 불필요
- 설정 변경 후 과거 급여 분쟁 → 확정 시점 스냅샷에 설정값 + 적용/미적용 사유까지 보존되어 있으므로 "당시 기준"으로 증명 가능

| 지표                     | Before                                  | After                                               |
|--------------------------|-----------------------------------------|-----------------------------------------------------|
| 급여 관련 문의           | 10건 중 1건 "왜 이 금액이냐"            | 0건                                                 |
| 문의 대응 시간           | 건당 30분+ (개발자가 DB 직접 추적)       | 고객이 명세에서 직접 확인                            |
| 과거 급여 근거 소급      | 불가 (현재 설정만 존재)                  | 확정 시점 스냅샷으로 보존                            |
| API 응답 정보            | 최종 금액 1개                           | 기본급, 수당 4종, 공제 5종, 적용/미적용 사유 포함     |
| 단위 테스트              | Spring 컨텍스트 필요 (수십 초)           | 순수 Java 20케이스 (3초)                             |
| 계산 오차 위험           | scale/RoundingMode 불일치 가능           | Money VO로 원천 차단                                 |

### 기술 스택

Spring Boot, JPA, DDD(Value Object/Domain Model), Caffeine Cache, Feign Client (서비스 간 통신)

| 판단                                    | 이유                                                                                                              |
|-----------------------------------------|-------------------------------------------------------------------------------------------------------------------|
