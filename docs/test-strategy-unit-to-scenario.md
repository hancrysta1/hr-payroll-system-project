# 테스트 전략 -- 단위->시나리오->CI 자동화 도입 과정

## 문제

### 테스트 부재로 인한 장애

초기에는 테스트 코드가 전혀 없었다. 신기능 도입이나 리팩토링을 진행할 때마다 기존 기능이 깨지는 일이 빈번했다.

대표적인 사례로, 급여 컨트롤러에서 웹(Manezy)과 앱(Crewezy)이 같은 API를 공유하도록 설계했는데, 사장/관리자/직원의 3단계 권한 체계와 지점의 급여 공개 설정(salaryVisibility: 0=비공개, 1=확정 후 공개, 2=항상 공개)이 맞물리면서 직원이 앱에서 급여를 조회할 때 사장 전용 접근 제어(`checkAdminAccess`)를 타게 되어 403 에러가 발생했다. 웹과 앱을 같은 엔드포인트로 효율적으로 돌리려던 의도였지만, 권한 분기 로직이 분리되지 않은 채 섞여 있었던 것이 원인이었다.

또한 회원가입 플로우에서도 문제가 있었다. 회원가입 시 지점을 생성하고 사용자와 매핑한 뒤 직원을 추가해야 하는데, 직원 추가는 사장 권한(adminLevel=2)만 허용된 API였다. 그런데 최초 가입 시점에는 JWT에 지점 정보가 없으므로 사장 권한이 부여되지 않은 상태였다. 이 문제는 JWT 리프레시 토큰을 활용해 가입 후 1차 발급 -> 지점 생성 및 유저 매핑 후 2차 발급하여 사장 권한을 다음 플로우(직원 추가)에 넘기는 방식으로 해결했다.

### 물리적 테스트 서버 시도와 한계

이러한 문제들이 실 배포 환경에 올라가다 보니, 별도의 테스트 서버 EC2 인스턴스를 구매하여 마이그레이션을 진행해보기도 했다. 하지만 이 경우 프론트엔드도 별도의 개발이 필요했고(테스트 서버 URI 추가), 서버 비용도 2배로 증가했다. 실서비스 매출이 발생하는 상황에서 지속 가능한 방법이 아니었다.

### 다운타임을 아무도 눈치채지 못한 사건

가장 크리티컬했던 건, 위 403 에러가 발생했을 때 기능이 동작하지 않는 동안 다운타임을 아무도 눈치채지 못했다는 것이었다. 보통은 배포 후 기능을 직접 돌려보는데, 급하게 배포하다 보니 이런 케이스를 고려하지 못한 것이 화근이었다. 실제 유저를 한 명이라도 모아야 하는 상황에서 이런 에러는 크리티컬했고, 이 허점을 테스트 코드를 통해 보완하고자 했다.

## 분석

### Phase 1. 도메인 단위 기능 테스트

처음에는 서비스/컨트롤러 단위의 기능 테스트를 작성하여 배포 전 CI에서 돌렸다.

- `SalaryCalculateServiceTest` -- 급여 계산 로직
- `AllowanceCalculatorTest` -- 가산수당 계산 (연장/야간/휴일)
- `MinimumWageServiceTest` -- 최저시급 로직
- `PayTypeWageValidatorTest` -- 급여 타입 유효성 검증

이 테스트들은 개별 기능의 정확성을 검증하는 데는 효과적이었지만, 기능과 기능 사이의 유기적인 흐름에서 발생하는 문제는 잡아주지 못했다. 예를 들어 급여 계산 로직 자체는 정상이지만, 사장이 설정을 변경한 후 직원이 조회할 때 권한 분기에서 에러가 나는 경우는 단위 테스트로는 발견할 수 없었다.

### Phase 2. 유저 플로우 기반 시나리오 테스트 도입

단위 테스트가 잡아주지 못하는 부분을 보완하기 위해, 실제 사용자가 웹과 앱에서 기능을 사용하는 플로우 순서대로 시나리오 테스트를 작성했다.

특히 민감한 급여 도메인의 경우, 다음과 같은 유저 케이스를 고려했다
- 사장(adminLevel=2) / 관리자(adminLevel=1) / 직원(adminLevel=0) 3단계 권한
- 지점 급여 공개 설정 3가지 (visibility: 0, 1, 2)
- 웹(Manezy)과 앱(Crewezy)의 접근 경로 차이

### MockitoExtension + MockMvc (standaloneSetup) 선택 이유

이 시나리오 테스트가 검증하려는 것은 "급여 계산이 맞는가"가 아니라 "사장/관리자/직원이 각각 API를 호출했을 때 권한과 설정에 따라 올바른 응답이 오는가"이다.

Crewezy(크루이지)와 Manezy(매니지)는 같은 백엔드를 바라보지만 사용자 역할이 다르다. 사장이 웹에서 급여를 확정하면, 직원은 앱에서 그 결과를 조회할 수 있어야 하고, 지점의 급여 공개 설정(salaryVisibility)에 따라 접근이 차단되기도 해야 한다. 이 권한 분기와 설정 분기 로직이 이번 테스트의 핵심 검증 대상이다.

실제로 필요한 것
- HTTP 요청이 컨트롤러에 도달하는 것
- `X-Branch-Admins` 헤더에서 권한 레벨을 올바르게 파싱하는 것
- 권한(사장/관리자/직원)과 설정(visibility 0/1/2)에 따른 분기 로직
- 올바른 상태 코드(200/403/404)와 에러 메시지 반환

필요하지 않은 것
- 실제 DB에서 급여 데이터를 읽어오는 것 (급여 계산 정확성은 단위 테스트 `SalaryCalculateServiceTest`에서 이미 검증)
- Eureka에 서비스를 등록하는 것
- 다른 마이크로서비스(branch-service 등)와 실제 통신하는 것

Spring context를 띄우면 이 "필요하지 않은 것들"이 전부 초기화되면서 DB 커넥션, Eureka 연결, Feign 클라이언트 등 외부 의존성 연결을 요구한다. 실제로 `@WebMvcTest`로 시도했을 때 Eureka/Cloud Config 연결 실패로 context 로딩 자체가 되지 않았다.

서비스 레이어를 Mock으로 대체하면, "이 유저의 급여는 50만원이다", "이 지점의 salaryVisibility는 2이다"처럼 정해진 값을 반환하도록 설정할 수 있다. DB에 실제 급여 데이터가 있을 필요가 없다.

이것이 가능한 이유는 테스트의 관심사가 분리되어 있기 때문이다
- 급여가 정확하게 계산되는가 -> 단위 테스트 (`SalaryCalculateServiceTest`, `AllowanceCalculatorTest`)
- 계산된 결과가 권한/설정에 따라 올바르게 노출되는가 -> 시나리오 테스트 (이번에 작성한 테스트)

시나리오 테스트는 후자에 집중하므로, 전자의 결과는 Mock이 대신 제공하면 된다. 이로써 DB 없이도, 외부 서비스 연결 없이도 실제 API 호출과 동일한 조건에서 유저 플로우를 검증할 수 있었다.

기술 스택
- MockitoExtension: JUnit 5와 Mockito를 연동하여 서비스/리포지토리 레이어를 Mock 객체로 대체
- MockMvc (standaloneSetup): 컨트롤러 객체만 직접 넣어 MockMvc를 구성. 실제 HTTP 요청/응답 파이프라인(JSON 직렬화, 상태 코드, 헤더 처리)은 그대로 검증하면서 외부 의존성은 완전히 제거
- @Mock + @InjectMocks: `@WebMvcTest`의 `@MockBean` 대신 사용. Spring context 로딩 없이 컨트롤러 레이어만 격리하여 테스트

### 유닛 테스트 vs 시나리오 테스트 역할 분담

| 구분 | 유닛 테스트 | 통합(시나리오) 테스트 |
|------|-----------|-------------------|
| 검증 대상 | 메서드/클래스 하나의 로직 | 여러 레이어가 협력하는 유저 플로우 |
| 외부 의존성 | 전부 Mock | 필요한 만큼만 Mock (또는 실제 연동) |
| 실행 속도 | 수 ms | 수백 ms ~ 수 초 |
| 실패 원인 파악 | 즉시 특정 가능 | 여러 지점을 추적해야 함 |
| 잡을 수 있는 버그 | 계산 오류, 분기 누락 | 권한 조합 오류, 레이어 간 데이터 불일치 |

```
유닛 테스트                              시나리오 테스트
─────────────                          ─────────────────
"급여가 정확하게 계산되는가"               "계산된 결과가 권한/설정에 따라
                                        올바르게 노출되는가"

SalaryCalculateServiceTest              SalaryOwnerFlowIntegrationTest
AllowanceCalculatorTest                 SalaryEmployeeFlowIntegrationTest
MinimumWageServiceTest                  WebLoginFlowTest
PayTypeWageValidatorTest                BranchOwnerFlowTest
```

유닛 테스트가 개별 부품의 정확성을 보장하고, 시나리오 테스트가 부품들이 조립된 후의 동작을 보장한다. 둘 중 하나만으로는 충분하지 않았다 -- 급여 계산 로직이 정확해도, 직원이 앱에서 조회할 때 사장 전용 권한 체크를 타면서 403이 터졌던 사례가 이를 증명한다.

## 해결

### 시나리오 테스트 구성

웹(Manezy) -- 사장 플로우

| 단계 | 서비스 | 테스트 | 설명 |
|------|--------|--------|------|
| 1 | user-service | `WebLoginFlowTest` | 회원가입 -> 로그인 -> JWT 발급 -> 프로필 조회 |
| 2 | branch-service | `BranchOwnerFlowTest` | 지점 조회 -> 직원 추가 -> 정보 수정 -> 급여 설정 -> 직원 삭제 |
| 3 | workpay-service | `SalaryOwnerFlowIntegrationTest` | 수당/공제 설정 -> 보험금 설정 -> 급여 조회 -> 개별 확정 -> 전체 확정 |

앱(Crewezy) -- 직원 플로우

| 단계 | 서비스 | 테스트 | 설명 |
|------|--------|--------|------|
| 1 | workpay-service | `SalaryEmployeeFlowIntegrationTest` | visibility별 접근 제어 (0=비공개, 1=확정 후 공개, 2=항상 공개) |

권한별 접근 제어 검증

| 권한 | 웹 (Manezy) | 앱 (Crewezy) |
|------|------------|-------------|
| 사장 (2) | 모든 기능 허용 | 403 -- "Manezy에서 확인해주세요." |
| 관리자 (1) | 403 -- "사장님만 접근 가능합니다." | visibility 설정에 따라 |
| 직원 (0) | 403 -- "사장님만 접근 가능합니다." | visibility 설정에 따라 |

### CI 파이프라인 연동

기존에는 workpay-service만 빌드 전 도메인 단위 테스트를 돌리고 있었고, 나머지 서비스는 `build -x test`로 테스트를 스킵하고 있었다. 각 서비스의 CI 단계에 시나리오 테스트를 추가했다.

```yaml
# user-service 변경 시
- name: Test user-service
  run: ./user-service/gradlew -p user-service test --no-daemon
    --tests 'com.epicode.integration.*'

# branch-service 변경 시
- name: Test branch-service
  run: ./branch-service/gradlew -p branch-service test --no-daemon
    --tests 'com.epicode.integration.*'

# workpay-service 변경 시 (기존 도메인 테스트 + 시나리오 테스트 추가)
- name: Test workpay-service
  run: ./workpay-service/gradlew -p workpay-service test --no-daemon
    --tests 'com.workpay.salary.domain.*'
    --tests 'com.workpay.salary.service.*Test'
    --tests 'com.workpay.salary.integration.*'
    --tests 'com.workpay.salary.controller.*'
```

CI에서 변경된 서비스만 감지하여 해당 서비스의 테스트만 실행하므로, 빌드 시간 증가는 경미하다 (Mock 기반이라 서비스당 5~10초 수준).

### 배포 단계와의 역할 분리

| 단계 | 역할 | 수행 위치 |
|------|------|----------|
| CI (PR 머지 전) | 코드 로직 검증: 단위 테스트 + 시나리오 테스트 | GitHub Actions |
| CD (배포 시) | 인프라 검증: Health Check + Eureka 등록 + 트래픽 모니터링 | EC2 (`deploy-canary.sh`) |

CI에서 코드 품질을, CD에서 인프라 안정성을 각각 검증하는 구조로, DB 없이 Mock으로 동작하는 시나리오 테스트는 CI에 배치하는 것이 적절하다.

### 배포 파이프라인 방어선

이전 -- 방어선 없는 파이프라인

```
코드 푸시 → build -x test → Docker 이미지 → EC2 docker-compose up -d → 끝
```

| 단계 | 검증 | 문제 |
|------|------|------|
| CI (빌드) | 컴파일만 통과하면 OK (`-x test`) | 로직 오류가 그대로 배포됨 |
| CD (배포) | `docker-compose up -d` 후 끝 | 컨테이너가 올라가기만 하면 성공 |
| 장애 감지 | 유저 신고 또는 직접 확인 | 다운타임을 아무도 모른 사건 발생 |

이후 -- 3단계 방어선

```
코드 푸시 → CI 테스트 (방어선 1) → Docker 이미지 → 카나리 배포 (방어선 2~3) → 전환
```

방어선 1. CI -- 코드 로직 검증 (PR 머지 전)

테스트 실패 시 PR 머지 자체가 차단되므로, 403 에러 같은 권한 오류가 프로덕션에 도달하는 것을 원천 차단한다.

방어선 2. CD -- 카나리 배포 (`deploy-canary.sh`)

| Phase | 검증 내용 | 실패 시 동작 |
|-------|---------|------------|
| Phase 1 | Docker 이미지 존재 확인 | 배포 중단 |
| Phase 2 | 새 버전 컨테이너 시작 (기존 포트 + 1000) | 배포 중단 |
| Phase 3 | Health Check (2분간 10초 간격, /actuator/health) | 자동 롤백 + 로그 출력 |
| Phase 4 | Eureka 등록 확인 (1분 대기) | 자동 롤백 + 디버깅 로그 |
| Phase 5 | 트래픽 모니터링 (1분간 구/신버전 50:50) | 자동 롤백 |
| Phase 6 | 구버전 제거 | -- |
| Phase 7 | 신버전을 정식 버전으로 태그 전환 | -- |

Phase 3~5에서 실패하면 자동 롤백한다. 구버전이 계속 트래픽을 처리하므로 다운타임 0.

방어선 3. 롤백 안전장치

카나리 배포에서 실패 시
1. 새 버전 컨테이너 자동 중지/삭제
2. canary 이미지 태그 삭제
3. 구버전이 중단 없이 계속 실행
4. 디버깅용 로그 자동 출력 (컨테이너 로그 tail -80)

```
                    CI 테스트          카나리 Health Check     카나리 모니터링
                    (방어선 1)          (방어선 2)              (방어선 3)
                    ──────────        ─────────────          ──────────────
잡는 것:            코드 로직 오류       앱 기동 실패             런타임 크래시
                    권한 분기 버그       DB 커넥션 실패           메모리 부족
                    API 응답 오류       환경변수 누락             Eureka 미등록
                    계산 공식 변경       포트 충돌               간헐적 오류

잡지 못하는 것:      인프라 문제         코드 로직 오류            저빈도 엣지 케이스
                    런타임 환경 차이     (이미 배포됨)            (모니터링 1분 내 미발현)
```

### 커버리지 변화

```
Phase 0        Phase 1              Phase 2
(테스트 없음)   (도메인 단위)          (+ 시나리오)

  0개      →     ~50개          →      ~90개

  검증 없음       계산 로직만            계산 + 권한 + 설정 + 플로우
                  ──────────           ─────────────────────────
                  "숫자가 맞나"          "유저가 쓸 때 문제없나"
```

핵심 지표
- 테스트 0개 -> 90개
- 권한/설정 조합 커버리지: 0% -> 12개 조합 전수 검증
- 배포 전 자동 검증: 없음 -> CI에서 PR 머지 전 필수 통과
- 수동 테스트 의존도: 100% -> 권한/급여 플로우는 자동화 완료

### 회고

잘한 점
- Mock 기반으로 DB/외부 서비스 의존 없이 빠르게 실행 가능한 테스트를 구성했다
- 실제 유저 플로우 순서대로 테스트를 배치하여, 기능 간 유기적 연결에서 발생하는 문제를 사전에 발견할 수 있게 했다
- CI에 통합하여 PR 머지 전 자동 검증 체계를 갖췄다

아쉬운 점
- 403 에러 발생 시 웹과 앱의 API를 분리(`detail` vs `web-detail`)하게 되면서 RESTful 원칙(같은 리소스 = 같은 URI)이 깨졌다. 접근 제어를 URL 분리가 아닌 메서드 내 분기나 인터셉터/필터 레벨에서 처리했다면 하나의 엔드포인트로 유지할 수 있었다.
- `@WebMvcTest`를 사용한 진정한 Spring 통합 테스트는 Eureka/Cloud Config 의존성 때문에 도입하지 못했다. 향후 테스트 프로파일 분리를 통해 개선할 여지가 있다.

배운 점
- 단위 테스트만으로는 서비스 간, 기능 간 연결에서 발생하는 문제를 잡을 수 없다
- 테스트 서버를 물리적으로 구축하는 것보다 Mock 기반 시나리오 테스트가 비용 효율적이다
- 배포 전 자동화된 테스트가 없으면 다운타임을 사람이 인지하기까지 시간이 걸린다
- 급하게 배포할수록 테스트가 더 중요하다
